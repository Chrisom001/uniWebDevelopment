Week 3 - APIs

I implemented an API to the Article database table by using the MVC framework to a degree by separating this off into another folder. Prior to creating the API I used procedural MYSQL queries to pull information from the database to the program. While working on the API, I was able to change this to an object orientated PDO style like what is below. Because the program itself only depended on the output, it meant the backend change had no affect on the front end.

function getArticleById($id) { global $pdo; $readArtByID = "SELECT * FROM GW_Articles WHERE articleID = '$id' LIMIT 1"; $readArticlesQuery = $pdo -> query($readArtByID); $readArticles = $readArticlesQuery -> fetchAll(PDO::FETCH_OBJ); return json_encode($readArticles); }

Using an API within a web based project is definitely beneficial as the main use is to make it easier to be able to change database or how the information is pulled from the database. This means that the change from MYSQL to PDO was much easier because the only thing I had to consider from the websites perspective was the data that it was expecting, rather than having to change entire sections of code on the site. It also makes the code cleaner as all of the SQL queries are stored within a single location and if you do have multiple sections of code which can use the same query, then you’re not repeating code which is not a positive thing when trying to write in a more object orientated way. It also means if someone is trying to find an error within the code, it can be easier to locate this. The final reason why the API is beneficial for the program is from a security point of view. It’s much easier to build checks and security into the system if the database isn’t tied directly to the program and it’s simply passing information between the two. This is because it is clearer when prepared statements are used, but also before passing information through to the API, it can be run through special character removers and other security checks.
